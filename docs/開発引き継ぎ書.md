## 1. 現在の結論

FishForecastMapは、糸島西岸〜唐津湾〜伊万里湾〜平戸周辺を対象とする、個人開発・個人利用の釣り向け参考情報Webアプリです。

- 釣り場地点マーカー、地点ポップアップ、釣果カード、魚種・エリア等のフィルタ。

- 承認済み魚種別名を、安定した内部魚種IDへ正規化する基盤。

## 2. 最重要の作業ルール

### 2.1 基本フロー

1. GitHub正本を確認する。
2. 次の作業を一つだけ整理する。
3. 1 Issue = 1 PRとする。
4. 実装担当と書き込み主体を明示する。
5. Codexを使う場合は、利用者の明示許可を待つ。
6. Codexへ渡す要件、対象外、完了条件、テスト条件を、最初の1回へ統合する。
7. 最新`main`から新規ブランチを作る。
8. PR作成後は最新HEAD、変更ファイル、全差分、CI、Vercel、制約違反を確認する。
9. 実ブラウザ確認が必要な場合はVercel Previewで確認する。
10. PRのマージは利用者が判断する。
11. マージ確認後に限り、利用者の明示指示に従ってIssueを`completed`として閉じる。
12. 大きな状態変化があれば、本リポジトリ内の`docs/開発引き継ぎ書.md`を更新する。

### 2.2 禁止事項

- 利用者の明示判断なしにCodexを起動しない。
- 利用者の明示判断なしにPRをマージしない。
- 利用者の明示判断なしにIssueを閉じない。
- 利用者の明示判断なしにremote DBへ適用しない。
- 複数の独立機能を同じIssue・PRへ混在させない。
- **同じPR、同じブランチ、同じCodex作業へ、分割したCodex指示を重ねない。**
- 同じPRへ2回目のトップレベル`@codex`指示を投稿しない。
- CI greenだけでマージ可能と判断しない。
- Codexの「完了」「PR更新」報告だけで実装成功と判断しない。
- 実施していないブラウザ確認、性能計測、外部送信、DB反映を完了扱いしない。
- Codex Cloud内のcommitや`make_pr`メタデータをGitHub反映済みと扱わない。
- 利用者本人が作成したPRをCodex作成と記録しない。
- GitHub上の対象ファイルを更新する依頼に対し、ローカルの別ファイルを出力して完了扱いしない。
- `main`へ直接ファイルを作成、更新、削除しない。
- force push、reset、履歴改変を利用者の明示許可なしに行わない。
- 不明な値をAI推測で`confirmed`にしない。
- placeholder URLや架空sourceを正式な根拠として保存しない。
- 未承認のバイナリファイルを作成、保存、変更、commitしない。

### 2.3 実行主体の記録

branch、commit、PRを誰が作成したかを分離して記録します。

- Codex Cloud: 隔離環境内の編集、テスト、commit候補。
- ChatGPTのGitHub接続: GitHub API経由で実際に行ったIssue、branch、commit、PR操作。
- 利用者: 利用者本人が手動で行ったpush、PR作成、merge、Issue close等。
- GitHub上で確認できない主体・経路: `未確認`。

GitHub上のbranch、HEAD SHA、PR番号、stateを取得するまで、GitHub反映完了と報告しません。

### 3.1 2026-07-21に発生した事象

Issue #204の初回実装PR #228に対し、最初の実装後にポップアップ視認性と単一ポップアップ制御を、別のCodex指示として同じPRへ重ねました。

その結果、PR #228では次の不具合が発生しました。

- `activePopupRef`の重複宣言。
- 実行文がMapLibreの設定オブジェクト内へ誤挿入。
- `useEffect`やcleanup内へ不完全な処理が混入。
- テストassertionの重複。
- CSSの重複。
- lint失敗。
- Vercel build失敗。

モック釣果削除も同じ作業へ重ねようとした際には、Codex側で作業対象の同期不整合が発生しました。

PR #228は未マージで閉じ、壊れたブランチとcommitを再利用せず、最新`main`からPR #232を作成して置き換えました。モック削除はIssue #229 / PR #233へ分離しました。

### 3.2 恒久ルール

> 同じPRへ分割したCodex指示を重ねない。

このルールは、次チャット以降も例外なく適用します。

#### Codex起動前

- 利用者の要望を一度整理する。
- 同一機能の完了に必要な要件を、一つの統合指示へまとめる。
- 対象ファイル、維持する既存機能、対象外、完了条件、テスト条件を明記する。
- 指示を分割して後から追加する前提でCodexを起動しない。

#### Codex起動後

- 同じIssue、PR、branchへ2回目のCodexタスクを起動しない。
- 同じPRへ追加の`@codex`コメントを投稿しない。
- CodexへGitHubコメントIDだけを渡さない。
- 同じ内容のトリガーを重複投稿しない。

#### PR作成後に追加要望が出た場合

追加要望を次のように分類します。

1. **別機能・追加スコープ**
   - 新しいIssueへ分離する。
   - 現PRへ追加しない。
   - 現PRのマージ後、最新`main`から新しいbranchとPRを作る。

2. **現PRの完了に必須だが、Codexへ分割指示が必要になる修正**
   - 追加Codex指示を重ねない。
   - 現PRを修復し続けるか、閉じて置換PRを作るかを利用者へ確認する。
   - コードの重複、不正挿入、同期不整合が少しでも疑われる場合は、現PRを閉じ、最新`main`からクリーンな置換PRを作る。
   - 壊れたbranch、commit、Codexワークスペースを再利用しない。

3. **人手で安全に直せる軽微な文書修正等**
   - 利用者の明示許可を得る。
   - Codexは追加起動しない。
   - 変更箇所と差分を限定する。

### 3.3 Codex完了後の必須確認

Codexが成功と報告しても、次を確認するまで完了扱いしません。

1. PRの最新HEAD SHA。
2. base branchとhead branch。
3. 変更ファイル一覧。
4. 全差分または全ファイルpatch。
5. 同じ宣言、同じ処理、同じtest、同じCSSの重複がないこと。
6. コードが構文上正しいscopeへ挿入されていること。
7. 対象外の変更がないこと。
8. 未承認バイナリがないこと。
9. lint。
10. typecheck。
11. test。
12. build。
13. `git diff --check`。
14. Vercel Preview。
15. DB変更がある場合はSupabase Migration PR Check。
16. 既存機能の回帰。

「Codexタスクが成功した」と「生成された実装が正しい」は別です。

### 3.4 CI失敗時

- 後続タスクを止める。
- 直ちに別のCodexタスクを重ねない。
- 失敗したworkflow、job、step、logを確認する。
- 一時障害かコード不具合かを区別する。
- コード不具合の場合は、現PR修正か置換PRかを利用者と決める。
- 同期不整合、重複挿入、不正scope混入がある場合は置換PRを優先する。

## 6. 地図・水深・環境予報

### 6.1 通常地図

- 国土地理院淡色地図。
- タイル: `https://cyberjapandata.gsi.go.jp/xyz/pale/{z}/{x}/{y}.png`
- `tileSize: 256`
- `maxzoom: 18`
- 航空写真、水深・3D地形、marker、popup、自動fitは独立して維持。

### 6.2 水深・3D地形

現在のsource:

- 第一source: `GEBCO_2026 Grid 15 arc-second`
- fallback: `NOAA NCEI ETOPO 2022 60 Arc-Second Bedrock`

主な機能:

- 2D水深色分け。
- 等深線。
- 陰影。
- MapLibre terrainによる3D表示。
- 高さ誇張。
- 視点preset。
- タップ地点の緯度経度、表示source、参考水深。
- GEBCO失敗時のETOPO fallback。
- 両方失敗時の通常地図fallback。

水深・3D地形は参考表示です。港内、岩礁、根、瀬、正確な海岸線を保証せず、航海・安全判断には使用できません。

### 6.3 Weather / Marine

Weather API:

- 気温。
- 天気コード。
- 降水量・降水確率。
- 風速・風向・突風。

Marine API:

- 海面水温。
- 潮位参考値。
- 波高・波向・波周期。
- 海流速度・海流方向。

条件:

- `timezone=Asia/Tokyo`
- `forecast_days=7`
- Weather / Marineを独立取得。
- 片方成功時はpartial表示。
- offsetなし時刻をAsia/Tokyoとして解釈する。
- time配列をfilterしてindexを詰めない。
- 全数値nullを取得成功と扱わない。
- API失敗、partial、古いcacheをSCORE低下へ直接使わない。
- 潮位参考値、波、風から「安全」と断定しない。

master data:

- `fish_species`
- `fish_species_aliases`
- `fishing_spots`
- `source_registry`

本人釣果:

- メールOTP / magic link login。
- 未login時はlocalStorage。

- Supabase `service_role` key。
- DB password / connection string。
- `.env.local`。
- 実token、実password、管理者権限値。

## 9. 外部釣り情報と地点調査のポリシー

- scraping、自動巡回、定期取得を行わない。
- 民間サイトの本文、画像、地図、魚種一覧、comment、profileを保存・転載しない。
- 民間サイトは地点候補や追加確認事項を知る入口に限定する。
- 公式・公的情報または複数の独立sourceで再確認する。
- 同じ原典・転載・同一運営元を独立sourceとして重複countしない。
- sourceが直接支えない属性を`supports`へ入れない。
- 一般立入、釣り可否、足場、駐車、トイレ、常夜灯を地点種別だけから推定しない。
- 魚種はobservedとexpectedを分離する。
- 不明値は`unknown / low / supportingSourceIds: []`を維持する。
- 外部サイトの地点DB、説明文、写真、評価、座標一覧をそのまま複製しない。
- 地点名、位置、属性は公的資料、地理院地図、自治体、港湾・漁港資料等から独自に整理する。
- 危険地点、立入禁止、小場所保護へ配慮する。

## 10. GitHub書き込みと失敗時のルール

### 10.1 GitHub書き込み

- 対象repository、Issue、PR、branch、pathを毎回確認する。
- read操作とwrite操作を区別する。
- `main`へ直接書き込まない。
- 文書更新依頼では、指定されたrepository内pathを更新する。
- 別のローカルfileやdownload用artifactを作って正本の更新と取り違えない。
- 書き込み成功応答だけで完了扱いせず、対象を再取得する。

### 10.2 2026-07-21の`main`誤操作記録

事故対応中に、`main`へ空file `__noop__`を誤作成し、直後に削除しました。

- 作成commit: `fb41dcf31b7440363aa248716b1c4dea29fbf3c2`
- 削除commit: `43942b63cc5d83e7a00e7493f53ff8c64aa7a0a6`

最終treeに`__noop__`は存在せず、実file差分は残っていません。履歴を消すためのforce pushやresetを行いません。

### 10.3 外部サービス書き込み失敗時

1. 後続作業を止める。
2. 未反映内容、影響範囲、現在の正本を固定する。
3. OpenAI側、connector側、外部service側、権限、入力内容を切り分ける。
4. 対象を再取得し、既に反映されていないか確認する。
5. 確認済み事実と原因仮説を分ける。
6. 再試行は原則1回。
7. 成功応答後も対象を再取得する。
8. 2回目も失敗した場合は、同じ操作を繰り返さず停止する。
9. 原因、影響範囲、暫定対応、恒久対策をまとめて報告する。
10. GitHub上で確認できるまで「反映済み」と報告しない。

## 11. テスト・確認

```bash
npm ci
npm run lint
npm run typecheck
npm test
npm run build
git diff --check
```

- `package.json`の`test`scriptを確認する。

- 静的文字列検索だけで要件を満たしたことにしていないか確認する。
- 最新HEADに対するCIか確認する。
- 古いcommitの成功statusで判断しない。
- CI、Vercel、Supabase Migration PR Checkが揃うまで待つ。
- 地図・UI変更では、Vercel PreviewでPC・smartphone表示、console error、failed requestを確認する。
- DB変更では、migration fileの存在とremote DBへの適用完了を分離して確認する。
