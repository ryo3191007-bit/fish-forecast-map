## 1. 現在の結論

FishForecastMapは、糸島西岸〜唐津湾〜伊万里湾〜平戸周辺を対象とする、個人開発・個人利用の釣り向け参考情報Webアプリです。

主な実装済み機能:

- Next.js / React / TypeScript / MapLibre GL JS。
- 通常地図、航空写真、水深・3D地形の3モード。
- 通常地図は国土地理院の淡色地図。
- 地点マーカー、ポップアップ、釣果カード、フィルタ。
- Open-Meteo Weather / Marineによる7日間の環境予報。
- Supabase master readと静的fallback。
- Supabase Auth、本人釣果の登録・編集・論理削除。
- GEBCO_2026 15秒を第一sourceとする参考水深・3D地形。
- 釣り場属性調査Schema v1.1.0と共通プロンプト。
- 唐津東港を対象としたChatGPT / Gemini / Claudeの3AI比較。

## 2. 最重要の作業ルール

### 2.1 基本フロー

1. 次の作業を一つだけ整理する。
2. GitHub Issueを作成する。
3. ユーザーがIssue内容を確認する。
4. 実装担当を決める。Codexを使う場合は、ユーザーの明示的な許可を待つ。
5. 1 Issue = 1 Draft PRとする。
6. 差分、CI、Vercel、制約違反をレビューする。
7. 実ブラウザ確認が必要な場合は、ユーザーがPreviewで確認する。
8. Draft解除・マージはユーザーが判断する。
9. マージ確認後、リンクされたIssueを`completed`として閉じる。

### 2.2 禁止事項

- ユーザーの許可なしにPRをマージしない。
- 複数の機能作業を同じIssue・PRへ混在させない。
- 外部情報の利用条件や公式回答を推測で確定しない。
- Codex管理中のPRブランチへ、Codex外からファイルcommitしない。
- CI greenだけでマージ可と判断せず、差分とテスト内容を確認する。
- 実施していないブラウザ確認、性能計測、外部送信を完了扱いしない。

### 2.3 Codex Cloud利用時の役割分離

2026-07-18の実地検証により、このリポジトリでGitHub Issueから`@codex`を起動する経路では、Codex Cloudの作業環境にGitHub HTTPS push用の認証情報が渡されていないことを確認しました。

この経路で確認できた範囲:

- Codex Cloudの隔離された作業環境内で、ファイル編集、テスト、commit作成は可能。
- 作業環境内のcommitは、そのままではGitHub上のcommitではない。
- `make_pr`のメタデータ生成は、GitHub上のPR作成完了を意味しない。
- Codexの完了コメント、Task URL、commit SHAの報告だけでは、branch・PRの存在証明にならない。
- GitHubへのbranch pushとDraft PR作成は、別の認証済み主体が行う必要がある。

Codex関連作業を完了扱いする前に、GitHub正本で次を確認します。

1. リモートbranchが実在する。
2. 報告されたHEAD SHAをGitHub上で取得できる。
3. PR番号、base、head、Draft状態が取得できる。
4. branch・PRを実際に作成した主体を確認する。
5. 変更ファイルとCIをGitHub上のHEADで確認する。

利用者本人またはChatGPTのGitHub接続がbranch・PRを作成した場合、Codexが作成したとは記録しません。実行主体が確認できない場合は「未確認」とします。

### 2.4 Codex CloudからGitHubへのPR作成検証結果（2026-07-18）

#### Issue #165で発生した事象

Codexは、10地点の調査ファイルを作成し、作業環境内でcommit `ca36105`を作成したと報告しました。また、`make_pr`を実行したと報告しました。

しかし、確認時点ではGitHub上に次が存在しませんでした。

- `ca36105`に対応するGitHub commit
- 対応するリモートbranch
- Issue #165を参照するDraft PR

したがって、この時点の成果は「Codex Cloud作業環境内の変更」であり、「GitHub反映済み」ではありません。

その後のPR #166は利用者本人が作成したものです。Codexが自動作成したPRとして扱いません。

#### Issue #167での再現試験

PR作成能力だけを確認するため、Issue #167で1ファイルのみを追加する再現試験を実施しました。

Codex Cloud内で成功した処理:

- `docs/diagnostics/CODEX_PR_CREATION_TEST.md`の作成
- `git diff --check`
- commit `be1331642c7631e8ab1a2010628d5ae1c0693459`の作成

失敗した処理:

```text
git push -u origin codex/issue-167-pr-creation-test
fatal: could not read Username for 'https://github.com': No such device or address
```

確認結果:

- GitHub上の対象branch: 存在しない
- GitHub上の対象PR: 存在しない
- `make_pr`メタデータ: 作成されたが、実PRとは扱わない

#### 原因の切り分け

直接原因は、Codex Cloudの作業用shellにGitHub HTTPS push用のユーザー名・token・credential helperがなく、`git push`が認証前に失敗したことです。

GitHub上でIssueコメントを投稿するBot接続と、Codex Cloud内の`git`コマンドが使用する認証経路は別です。BotがIssueへコメントできても、作業用shellがGitHubへpushできることの証明にはなりません。

GitHub側のリポジトリ権限不足やbranch protectionが原因ではありません。今回のエラーは、それらの判定へ到達する前の認証情報取得段階で発生しました。

#### 固定運用

- 現在のIssue `@codex`経路では、新規branchのpush・新規PR作成をCodexへ任せない。
- Codexを利用する場合も、成果は作業環境内の編集・テスト・commit候補として扱い、GitHub反映主体を別途明示する。
- branch・ファイル・PRをChatGPTのGitHub接続で作成した場合は、ChatGPTによる操作として記録する。
- 利用者がPRを作成した場合は、利用者作成と明記する。
- GitHub上のbranch、HEAD SHA、PR番号を取得するまで完了報告しない。
- すぐ確認できる実行結果を、定期確認タスクだけに任せて1時間後へ先送りしない。
- 認証経路が変更された場合も、最小の専用Issueで再検証してから運用を変更する。

## 6. 環境予報とSCORE

Weather API:

- 気温、天気コード、降水量、降水確率、風速、風向、突風。

Marine API:

- 海面水温、潮位参考値、波高、波向、波周期、海流速度、海流方向。

条件:

- `timezone=Asia/Tokyo`
- `forecast_days=7`
- Weather / Marineを独立取得。
- 片方成功時はpartial表示。
- 30分以内のcacheはfresh。
- 24時間以内のcacheはAPI失敗時fallback。
- 24時間超は使用しない。

環境値は現在のSCOREへ本格反映していません。次のSCOREは、地点固定特性、魚種生態、季節、環境、本人釣果を分離し、加点・減点理由を説明できるルールベース方式から設計します。

## 7. Supabase / DBの状態

master data:

- `fish_species`
- `fishing_spots`
- `source_registry`

未設定、取得エラー、空データ時は静的データへfallbackします。

本人釣果:

- メールOTP / magic linkログイン。
- 未ログイン時はlocalStorage。
- ログイン済みかつDB利用可能時はSupabaseへ保存。
- DB失敗時はlocalStorage fallback。
- owner-scoped。
- 削除は論理削除。
- 旧localStorage記録は自動移行せず、選択した記録だけ明示移行。

新しいDB変更は`supabase/migrations/`を正本とします。旧SQLファイルをmigration代わりに使用しません。
