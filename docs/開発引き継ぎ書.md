## 2. 最重要の作業ルール

### 2.1 基本フロー

1. GitHub正本を確認する。
2. 次の作業を一つだけ整理する。
3. 1 Issue = 1 PRとする。
4. 実装担当と書き込み主体を明示する。
5. Codexを使う場合は、利用者の明示許可を待つ。
6. Codexへ渡す要件、対象外、完了条件、テスト条件を、最初の1回へ統合する。
7. 最新`main`から新規ブランチを作る。
8. PR作成後は最新HEAD、変更ファイル、全差分、CI、Vercel、制約違反を確認する。
9. 実ブラウザ確認が必要な場合はVercel Previewで確認する。
10. PRのマージは利用者が判断する。
11. マージ確認後に限り、利用者の明示指示に従ってIssueを`completed`として閉じる。
12. 大きな状態変化があれば、本リポジトリ内の`docs/開発引き継ぎ書.md`を更新する。

### 2.2 禁止事項

- 利用者の明示判断なしにCodexを起動しない。
- 利用者の明示判断なしにPRをマージしない。
- 利用者の明示判断なしにIssueを閉じない。
- 利用者の明示判断なしにremote DBへ適用しない。
- 複数の独立機能を同じIssue・PRへ混在させない。
- **同じPR、同じブランチ、同じCodex作業へ、分割したCodex指示を重ねない。**
- 同じPRへ2回目のトップレベル`@codex`指示を投稿しない。
- CI greenだけでマージ可能と判断しない。
- Codexの「完了」「PR更新」報告だけで実装成功と判断しない。
- 実施していないブラウザ確認、性能計測、外部送信、DB反映を完了扱いしない。
- Codex Cloud内のcommitや`make_pr`メタデータをGitHub反映済みと扱わない。
- 利用者本人が作成したPRをCodex作成と記録しない。
- GitHub上の対象ファイルを更新する依頼に対し、ローカルの別ファイルを出力して完了扱いしない。
- `main`へ直接ファイルを作成、更新、削除しない。
- force push、reset、履歴改変を利用者の明示許可なしに行わない。
- 不明な値をAI推測で`confirmed`にしない。
- placeholder URLや架空sourceを正式な根拠として保存しない。
- 未承認のバイナリファイルを作成、保存、変更、commitしない。

### 2.3 実行主体の記録

branch、commit、PRを誰が作成したかを分離して記録します。

- Codex Cloud: 隔離環境内の編集、テスト、commit候補。
- ChatGPTのGitHub接続: GitHub API経由で実際に行ったIssue、branch、commit、PR操作。
- 利用者: 利用者本人が手動で行ったpush、PR作成、merge、Issue close等。
- GitHub上で確認できない主体・経路: `未確認`。

GitHub上のbranch、HEAD SHA、PR番号、stateを取得するまで、GitHub反映完了と報告しません。

- Codex作業対象の同期不整合。

### 3.2 恒久ルール

> 同じPRへ分割したCodex指示を重ねない。

Codex起動前:

- 利用者の要望を一度整理する。
- 同一機能の完了に必要な要件を、一つの統合指示へまとめる。
- 対象ファイル、維持する既存機能、対象外、完了条件、テスト条件を明記する。
- 後から追加指示を重ねる前提でCodexを起動しない。

PR作成後に別scopeが出た場合:

- 別機能・追加scopeは新しいIssueとPRへ分離する。
- 現PRの完了に必須だが同期不整合や重複挿入が疑われる場合は、置換PRを優先する。
- 軽微な文書修正を人手で行う場合も、利用者の明示許可と限定差分を必要とする。

### 3.3 Codex完了後の必須確認

1. PRの最新HEAD SHA。
2. base branchとhead branch。
3. 変更ファイル一覧。
4. 全差分または全ファイルpatch。
5. 宣言、処理、test、CSSの重複がないこと。
6. コードが構文上正しいscopeへ挿入されていること。
7. 対象外の変更がないこと。
8. 未承認バイナリがないこと。
9. lint。
10. typecheck。
11. test。
12. build。
13. `git diff --check`。
14. Vercel Preview。
15. DB変更がある場合はSupabase Migration PR Check。
16. 既存機能の回帰。

「Codexタスクが成功した」と「生成された実装が正しい」は別です。

## 4. 地点評価・地図の相互導線

Issue #204 / PR #232で地点評価から地図への一方向フォーカスを実装し、Issue #243 / PR #244で地図から地点評価への明示的な逆方向導線を追加しました。

1. 地点名コンボボックス右の`Map`ボタンを押す。
2. 地図セクションへスクロールする。
3. 選択地点へパン・ズームする。
4. 対象地点マーカーを強調する。

### 4.2 地図から地点評価

1. 地点マーカーのポップアップを開く。
2. `地点評価`ボタンを押す。
3. 対象地点を地点評価の選択地点へ設定する。

6. 選択済みの予報日時は明示的にリセットしない。

- 地点変更だけで地図を自動移動しない。

- popup / marker参照をcleanupする。
- 地点ポップアップはコンパクトな白背景・濃い文字・モバイル最大幅を維持する。

## 5. 地点評価UIと地点詳細表示

### 5.1 タブと日時操作

Issue #243 / PR #244で次を実装済みです。

- タブ順は`環境 / 釣場 / 地形 / 評価`。
- 初回表示は`環境`。
- 地点変更や通常の再レンダーで、利用者が選択中のタブを強制リセットしない。
- 日付・時刻は2列表示。
- 時間操作は`< / 現在時刻 / >`の1行表示。
- 視覚表示を短縮しても、`aria-label`は説明的な名称を維持する。
- 地点評価ヘッダーの地域名重複表示は削除。
- 地点名コンボボックス内の地域名検索・候補表示は維持。
- `Map`ボタンは地点名入力欄の右へ配置。

全魚種評価の遷移・history/hash復元と、必要な明示的`評価`タブ切替を壊さないでください。

### 5.2 地点詳細の表示分類

Issue #237 / PR #240で、全28地点に次の分類を適用済みです。

- `海底・沿岸地形`: 砂地、岩礁、藻場、河口等の自然地形。
- `釣り場の構造・足場`: 堤防、岸壁、テトラ、磯、砂浜等。
- `波止`は通常UIで`堤防`へ統一。
- 単独の地点種別や長い経路説明を通常UIへ重複表示しない。
- raw evidence、source metadata、confidence、内部noteは監査用に保持し、通常UIへ直接露出しない。

### 5.3 暫定表現の除去

Issue #238 / PR #241とIssue #239 / PR #242で、利用者向け表示を整理済みです。

除去対象の例:

- `候補（現行未確認）`
- `候補（low / medium / high）`
- 末尾の`候補`
- 末尾の`（low / medium / high）`
- 全角・半角括弧、空白違い

表示例:

- `照明候補（現行未確認）` → `照明`
- `頭上電線注意候補（low）` → `頭上電線注意`
- `手すり候補（low）` → `手すり`

重要:

- 調査元データを削除せず、通常UIだけを整形する。
- `信憑性: 低`等の利用者向けconfidence表示は残す。
- `なし`、`利用不可`等の否定値を肯定表示へ変換しない。
- 暫定表現しか残らない値は、事実を捏造せず`調査済み・未確定`へフォールバックする。
- 検索UIの正常な`候補がありません`は削除しない。

### 6.3 Weather / Marine

Weather API:

- 気温。
- 天気コード。
- 降水量・降水確率。
- 風速・風向・突風。

Marine API:

- 海面水温。
- 潮位参考値。
- 波高・波向・波周期。
- 海流速度・海流方向。

条件:

- `timezone=Asia/Tokyo`
- `forecast_days=7`
- Weather / Marineを独立取得。
- 片方成功時はpartial表示。
- offsetなし時刻をAsia/Tokyoとして解釈する。
- time配列をfilterしてindexを詰めない。
- 全数値nullを取得成功と扱わない。
- API失敗、partial、古いcacheをSCORE低下へ直接使わない。
- 潮位参考値、波、風から「安全」と断定しない。

## 9. Supabase / DB

master data:

- `fish_species`
- `fish_species_aliases`
- `fishing_spots`
- `source_registry`

本人釣果:

- メールOTP / magic link login。
- 未login時はlocalStorage。
- login済みかつDB利用可能時はSupabaseへ保存。
- DB失敗時はlocalStorage fallback。
- owner-scoped。
- 削除は論理削除。
- 旧localStorage記録は自動移行せず、選択した記録だけ明示移行。

新しいDB変更は`supabase/migrations/`を正本とします。旧`supabase/sql/`をmigration代わりに使用しません。

remote DB操作は、利用者の明示指示なしに行いません。リポジトリへmigrationがmergeされたことと、本番DBへ適用済みであることを分離して記録します。

GitHub、Issue、PR、チャット、ログへ次を貼りません。

- Supabase `service_role` key。
- DB password / connection string。

## 12. GitHub書き込みと失敗時のルール

### 12.1 GitHub書き込み

- 対象repository、Issue、PR、branch、pathを毎回確認する。
- read操作とwrite操作を区別する。
- `main`へ直接書き込まない。
- 文書更新依頼では、指定されたrepository内pathを更新する。
- 別のローカルfileやdownload用artifactを作って正本の更新と取り違えない。
- 書き込み成功応答だけで完了扱いせず、対象を再取得する。

### 12.2 2026-07-21の`main`誤操作記録

事故対応中に、`main`へ空file `__noop__`を誤作成し、直後に削除しました。

- 作成commit: `fb41dcf31b7440363aa248716b1c4dea29fbf3c2`
- 削除commit: `43942b63cc5d83e7a00e7493f53ff8c64aa7a0a6`

最終treeに`__noop__`は存在せず、実file差分は残っていません。履歴を消すためのforce pushやresetを行いません。

### 12.3 外部サービス書き込み失敗時

1. 後続作業を止める。
2. 未反映内容、影響範囲、現在の正本を固定する。
3. OpenAI側、connector側、外部service側、権限、入力内容を切り分ける。
4. 対象を再取得し、既に反映されていないか確認する。
5. 確認済み事実と原因仮説を分ける。
6. 再試行は原則1回。
7. 成功応答後も対象を再取得する。
8. 2回目も失敗した場合は、同じ操作を繰り返さず停止する。
9. 原因、影響範囲、暫定対応、恒久対策をまとめて報告する。
10. GitHub上で確認できるまで「反映済み」と報告しない。
