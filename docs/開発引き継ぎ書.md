## 2. 最重要の作業ルール

### 2.1 基本フロー

1. ChatGPTが次の作業を一つだけ整理する。
2. GitHub Issueを作成する。
3. ユーザーが内容を確認する。
4. Codexを使う場合は、ユーザーの明示的な`Codex起動OK`を待つ。
5. Codex Botの👀、返信、Task URLのいずれかを確認してから「起動済み」と判断する。
6. PR作成後はChatGPTが差分・CI・Vercel・制約違反をレビューする。
7. 最終マージはユーザーが行う。
8. マージ確認後、ChatGPTが対応Issueを`completed`で閉じる。

### 2.2 絶対に守ること

- 必要な作業は一つずつ提示する。
- ユーザーの完了報告前に次作業を重ねない。
- PRをユーザーの代わりにマージしない。
- Codex起動前に親コミット、既存PR継承可否、想定消費を確認する。
- Codex外からCodex管理中のPRブランチへファイルcommitすると同期不能になる場合がある。
- Codexへ「最新コメントを見て」だけを渡さず、修正要件を全文で渡す。
- CI greenだけでマージ可と判断せず、テストが実装本体を実行しているか確認する。
- ユーザーの明示許可なしにGitHubコメント、コミット、Issue/PR操作を行わない。

### 2.3 Codexで既存PRを追加修正する確定手順

既存PRの追加修正は、PR Conversationのトップレベルコメントに`@codex`と修正要件全文を投稿して起動します。PR Review submissionに`@codex`を書いただけでは起動しなかったため、レビュー記録と起動コメントは分けます。リンク元Issueから起動すると新しいタスク・ブランチ・PRとして開始されるため、既存PRの追加修正には使いません。

必須手順:

1. 既存PRのstate、head branch、head SHAを確認する。
2. Codex管理ブランチへCodex外のファイルcommitが入っていないか確認する。
3. 修正内容をPR reviewまたは通常コメントとしてPRへ記録する。
4. ユーザーの`Codex起動OK`を待つ。
5. PR Conversationのトップレベルコメントへ`@codex`と修正要件全文を投稿する。
6. Codex Botの👀・返信・Task URLのいずれかを確認し、初めて「起動済み」とする。
7. PRのHEAD SHA更新、追加commit、完了コメントを確認し、初めて「更新済み」とする。
8. 起動しない、または更新できない場合は成功例との差分を切り分け、ユーザー確認なしにPRを閉じたり新規PRへ切り替えたりしない。

禁止事項:

- 既存PR修正をリンク元Issueから起動する。
- `@codex`をPR Review submissionだけに書いて起動済みと判断する。
- Bot反応なしで「起動済み」と報告する。
- 👀だけで修正完了扱いにする。
- 原因未確認のままPRを閉じる、または置き換えPRを作る。
- Codex管理ブランチへCodex外からファイルcommitする。
- Draft状態が原因だったという仮説を、検証済み事実として扱う。

補足: Issue側タスクへ既存PRブランチ名を指定しても、その作業環境に対象remote/branchがなければ取得できません。PR本文・タイトル・コメント等のmetadata更新と、PRブランチへのファイルcommitは区別してください。

## 4. Supabase / DBの状態

### 4.1 master data

Supabaseから次を読み取ります。

- `fish_species`
- `fishing_spots`
- `source_registry`

未設定、取得エラー、空データ時は静的データへfallbackします。

### 4.2 Auth / 自分の釣果記録

- メールOTP / magic linkログイン。
- 未ログイン時はlocalStorage。
- ログイン済みかつDB利用可能時はSupabaseへ保存。
- DB失敗時はlocalStorage fallback。
- DB行はowner-scoped。
- 削除は`is_deleted = true`の論理削除。
- 旧localStorage記録は自動移行せず、ユーザーが選択した記録だけ明示移行。
- 内部型・テーブル名に`ExternalCatchMemo` / `external_catch_memos`が残るのは後方互換のため。

### 4.3 migration

新しいDB変更の正本:

```txt
supabase/migrations/<timestamp>_<description>.sql
```

現在、手動実行すべきSQLはありません。`supabase/sql`の旧設計ファイルを新規migration代わりに使用しないでください。

Marine API:

条件:

- `timezone=Asia/Tokyo`
- `forecast_days=7`
- Weather / Marineを独立取得
- 片方成功時はpartial表示
- 30分以内のcacheはfresh
- 24時間以内のcacheはAPI失敗時fallback
- 24時間超は使用しない
- offsetなし時刻はAsia/Tokyoとして解釈
- 地点切替時はAbortと`isActive`で古い応答を防止

- TID values SHA: `f39a3d090f387d124c1b5a10ecfff113f186b5f916ad2cc4001d5bebf2a70688`

### 6.2 表示・fallback

- GEBCO成功: GEBCO color / hillshade / contour / label / terrain。
- GEBCO metadata、tile、decode、terrain初期化失敗: ETOPOへ切替。
- ETOPOも失敗: 水深layer、terrain、海岸線ライン、陸地マスク、出典を解除し通常地図へ戻す。
- 同じerrorはdedupe。
- 遅れて届いた非表示sourceのerrorで状態を飛ばさない。
- 通常地図復帰時の表示: `水深データを読み込めなかったため通常地図へ戻しました`。

### 6.3 TID UI

- `public/bathymetry/gebco-2026/tid-crop.json`を自サイト内から読み込む。
- `moveend`で現在centerを再集計。
- 中心周辺17×17セル。
- nodata `127`を割合の分母から除外。
- bounds外、読込失敗、nodata-onlyは非致命的な案内。

### 6.4 海岸線ライン・陸地マスク出典

現行の海岸線補助は外部GSI overlayを使用しません。`npm run generate:bathymetry`で、GEBCO 15秒DEMの非負標高セルから完全不透明の落ち着いた緑の陸地マスクを生成し、0m境界から海岸線ラインを生成します。出典はGEBCO正本の出典に含め、航海・安全判断不可の注記を維持します。

### 6.5 生成・変換

通常生成:

```bash
npm run generate:bathymetry
```

公式NetCDFから正本を更新する手動変換:

```bash
python -m pip install netCDF4
node tools/bathymetry/convert-gebco-netcdf.mjs /local/gebco.nc /local/gebco_tid.nc
```

runtime/buildから外部GEBCO/NOAA取得は行いません。人工的な`Math.sin(...)*12`等の起伏を絶対に追加しません。


### 6.6 Post-MVP-045の現行海岸線構成

Issue #129 / PR #132で、Post-MVP-038当時のGSI標準地図overlay構成を置き換えました。現在はGEBCO由来の海岸線ラインと完全不透明の緑の陸地マスクを自サイト内の静的assetとして配信し、外部GSI overlayを使用しません。PR #132はmergedで、merge commitは `f6c8453b8d55ee3b12bb862766f07408921ed082` です。

### 6.7 高精細3D海底地形ビュー方針

Issue #133 / Post-MVP-046で、将来の高精細3D海底地形ビューに向けた正本 `docs/HIGH_RESOLUTION_3D_BATHYMETRY_PLAN.md` を追加しました。現行GEBCO/ETOPOで改善するPhase A、条件を満たす具体的データセットを確認できた場合だけ1海域限定の高精細PoCへ進むPhase B、多解像度3D本番化のPhase C、魚種・季節・適性レイヤーを保留するPhase Dに分けます。ライセンス不明データや有料データ、スクレイピング、巨大ファイルcommit、DB変更は引き続き対象外です。

### 6.8 Post-MVP-047の水深3D表示操作改善

Issue #135ではPost-MVP-046 Phase Aの最初の実装として、GEBCO_2026／ETOPO fallback、海岸線ライン、緑の陸地マスク、TID表示、既存fallbackを維持しつつ、水深・3D地形パネルへ高さ誇張スライダー、現在倍率表示、1.0×リセット、真上・斜め・低角度の視点プリセットを追加しました。高さ誇張は1.0〜4.0の表示上の強調に限定し、水深データ精度が上がるものではない注記を追加しています。スライダー変更はterrain exaggerationだけを更新し、center・zoom・pitch・bearingを変更しません。
