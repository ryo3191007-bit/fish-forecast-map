## 2. 最重要の作業ルール

### 2.1 基本フロー

1. GitHub正本を確認する。
2. 次の作業を一つだけ整理する。
3. 1 Issue = 1 Draft PRとする。
4. 実装担当を明示する。
5. 差分、CI、Vercel、制約違反、事実根拠をレビューする。
6. 実ブラウザ確認が必要な場合はPreviewで確認する。
7. Draft解除・マージは利用者が判断する。
8. マージ確認後に限り、リンクされたIssueを`completed`として閉じる。

### 2.2 禁止事項

- 利用者の明示判断なしにPRをマージしない。
- 複数の機能作業を同じIssue・PRへ混在させない。
- 外部情報、利用条件、公式回答、実行主体を推測で確定しない。
- CI greenだけでマージ可と判断しない。
- 実施していないブラウザ確認、性能計測、外部送信、DB反映を完了扱いしない。
- Codex Cloud内のcommitや`make_pr`メタデータをGitHub反映済みと扱わない。
- 利用者本人が作成したPRをCodex作成と記録しない。
- 即時確認できる結果を、定期確認だけに任せて先送りしない。
- 不明な値をAI推測で`confirmed`にしない。
- placeholder URLや架空sourceを正式な根拠として保存しない。

### 2.3 実行主体の記録

branch、commit、PRを誰が作成したかを分離して記録します。

- Codex Cloud: 隔離環境内の編集、テスト、commit候補。
- ChatGPTのGitHub接続: GitHub API経由で実際に行ったbranch、commit、PR操作。
- 利用者: 利用者本人が手動で行ったpush、PR作成、Draft解除、merge等。
- GitHub上で確認できない主体・経路: `未確認`。

GitHub上のbranch、HEAD SHA、PR番号、Draft状態を取得するまで、GitHub反映完了と報告しません。

## 8. Supabase / DBの状態

master data:

- `fish_species`
- `fish_species_aliases`
- `fishing_spots`
- `source_registry`

既存master dataは未設定、取得エラー、空データ時に静的データへfallbackします。魚種別名の空データだけは下記のとおりfallbackしません。

魚種別名の正規化（Issue #196 / PR #212）:

- 魚種本体の `fish_species.id` は、`aji`、`chinu` など人が読める安定したtext IDを維持する。別名行の `fish_species_aliases.id` はUUIDとする。
- 入力はNFKC正規化 → 前後空白除去 → 小文字化した `match_key` で完全一致させる。部分一致や類似度による推測は行わない。
- resolverは `approval_status = 'approved'` かつ `is_active = true` の行だけを解決対象とする。approvedかつactiveな `match_key` は全魚種で一意とする。
- `anon` / `authenticated` には承認済みかつ有効な行の `select` だけを許可し、write権限は付与しない。
- `チヌ`、`黒鯛`、`クロダイ` はすべて `chinu` として解決し、一覧検索、魚種別件数集計、外部メモ加点、SCORE v2の釣果履歴、SCORE v2の直接対象魚種根拠で同一魚種として扱う。
- 未登録名と空文字は `unresolved`、複数魚種候補は `conflict` としてfail-closedにする。推測で既存魚種へ結び付けず、検索一致、件数、加点、採点の対象外とする。
- Supabase取得成功で別名が0件の場合は空配列を正とする。Supabase障害、未設定、またはテーブル未作成による取得エラー時だけstatic fallbackを使う。
- 初期に追加する別名は `黒鯛`、`クロダイ` に限定する。正式名 `チヌ` も同じresolver経路を通し、その他の追加候補はIssue #211で扱う。
- migrationファイル `supabase/migrations/20260720150000_add_fish_species_aliases.sql` はmainへmerge済み。ただし、リポジトリへのmergeとremote Supabase本番DBへの実適用完了は別であり、実適用は未確認のため完了と断定しない。
- PR #212の最新HEAD `911d14bc400a65464df4b671daefc068dc654cb0` でCI、Supabase Migration PR Check、Vercelはすべて成功済み。

本人釣果:

- メールOTP / magic linkログイン
- 未ログイン時はlocalStorage
- ログイン済みかつDB利用可能時はSupabaseへ保存
- DB失敗時はlocalStorage fallback
- owner-scoped
- 削除は論理削除
- 旧localStorage記録は自動移行せず、選択した記録だけ明示移行

新しいDB変更は`supabase/migrations/`を正本とします。旧SQLファイルをmigration代わりに使用しません。

GitHub、Issue、PR、チャット、ログへ次を貼りません。

- `.env.local`
- 実トークン、実パスワード、管理者権限値

## 14. 外部サービス書き込み失敗時の対応ルール（2026-07-19追記）

### 14.1 今回発生した事象

Issue #193への確定仕様コメント追加時に、GitHubへ送信される前のOpenAI側の安全判定で2回ブロックされた。

確認できたこと:

- 同じIssue・同じコメント追加機能は直前まで正常に動作していた。
- GitHubのHTTPエラー、権限エラー、Issue状態エラーは返っていない。
- 失敗後もGitHubの読み取りは正常だった。
- 正確な内部判定理由は開示されていないため、根本原因は断定できない。
- 「安全情報が欠けていても評価を続ける」という表現が、安全制御を弱める指示として誤解された可能性が高い。これは確認済み事実ではなく原因仮説として扱う。

再試行では、仕様内容を変えずに処理条件を技術的に明確化した。

- 欠損項目は判定対象外とする。
- 欠損項目を安全確認済みとは扱わない。
- 取得済み項目は個別に危険閾値を判定する。
- 危険条件が検出されず、SCORE v2のcoverage要件を満たす場合だけ通常計算へ進む。

この表現でIssue #193への追加に成功した。

- 成功コメントID: `5015082382`
- 成功後にIssueを再取得し、コメント数が16件から17件へ増えたことを確認済み。
- Codex起動・コード実装は行っていない。

### 14.2 今後の必須対応

外部サービスへの書き込み、CI、コネクタ、GitHub操作等で一度でも問題が発生した場合は、次の順で対応する。

1. 後続作業と新しい仕様確認をいったん止める。
2. 未反映の内容、影響範囲、現在の正本を固定する。
3. OpenAI側、コネクタ側、外部サービス側、権限、入力内容のどこで失敗したかを切り分ける。
4. 対象を再取得し、実は既に反映されていないか、同等内容が既存記録にないかを確認する。
5. 原因が断定できない場合は、確認済み事実と原因仮説を分離して報告する。
6. 再試行は原則1回とし、仕様の意味を変えず、誤解されにくい技術表現へ直す。
7. 書き込み成功の応答だけで完了扱いせず、対象を再取得してID、本文、件数、状態等を確認する。
8. 2回目も失敗した場合は同じ確認を利用者へ繰り返さず、手動貼付用の確定文面を提示する。
9. 原因、影響範囲、暫定対応、恒久対策を一組で報告する。
10. GitHub上で確認できるまで「記録済み」「反映済み」と報告しない。

この対応は、システム仕様の正しさだけでなく、確認工数と手戻りを減らして開発を円滑に進めるための必須運用とする。
