## 1. プロジェクト概要

FishForecastMap は、釣行判断を支援する個人向けの釣り予報マップアプリです。

主な機能・方針は以下です。

- Next.js / React / TypeScript ベースのWebアプリ。
- MapLibre GL JS で地図表示。
- 国土地理院航空写真タイルを利用。
- Open-Meteo Weather / Marine API を利用。
- Supabase はマスターデータ読み取り、将来的な外部釣果メモ保存に利用。
- 本番アプリではマスターデータをSupabaseから読み取る状態まで確認済み。
- 外部釣果メモは、現時点ではまだ `localStorage` 保存。
- 外部メモDB保存は、Auth/RLSの安全な土台を作っている途中。

重要なドメイン方針:

- `sea_level_height_msl` は、あくまで「潮位に似た参考値」。公式潮汐表・安全判断・航行判断には使わない。
- 外部釣果情報のスクレイピング、自動収集、定期実行ジョブ、AI解析は行わない。
- 外部メモはユーザーが手動で入力する。
- 外部サイト本文の転載や自動取り込みは行わない。
- 現時点で `source_registry` に `crawlPolicy === "allowed"` のサイトはない。

## 2. 最重要の運用ルール

このプロジェクトは、GitHub Issue / PR を中心に、ChatGPT と Codex を分担して進めています。

### 2.1 役割分担

- ユーザー: 最終判断、PRマージ、手動設定、手動SQL実行。
- ChatGPT: 方針整理、Issue作成、Codex起動コメント、PRレビュー、CI確認、Issue完了クローズ、手動作業案内。
- Codex: GitHub Issue の指示に従って実装PRを作成。

### 2.2 基本フロー

1. ChatGPTが次の作業内容を整理する。
2. GitHub Issueを作成する。
3. ユーザーが内容を確認する。
4. ユーザーが明示的に `Codex起動OK` と送る。
5. ChatGPTがIssueへCodex起動コメントを投稿する。
6. CodexがPRを作成する。
7. ユーザーがPR URLをChatGPTへ渡す。
8. ChatGPTがPR差分・CI・制約違反をレビューする。
9. 問題なければChatGPTがPRにレビューコメントを投稿する。
10. ユーザーがPRをマージする。
11. ChatGPTがマージ済みを確認する。
12. ChatGPTが対応Issueを `completed` でクローズする。

### 2.3 絶対に守ること

- 作業は一つずつ進める。
- ユーザーの完了報告前に、次の作業を重ねて出さない。
- Codexを起動する前に、必ずユーザーの `Codex起動OK` を待つ。
- ChatGPTはPRをマージしない。PRマージは必ずユーザーが行う。
- ChatGPTは、マージ確認後にIssueを `completed` でクローズする。
- 手動SQL実行、Vercel環境変数設定、Supabase Dashboard設定はCodexにやらせない。
- SQLを実DBに反映する時は、必ず「手動チェックポイント」としてユーザーに案内する。

## 3. セキュリティ・秘密情報ルール

このプロジェクトでは、秘密情報をGitHub・Issue・PR・チャットへ貼らないことを徹底します。

### 3.1 絶対に貼らないもの

- Supabase `service_role` key
- Supabase DB password
- Supabase DB URL / Postgres connection string
- `.env.local`
- 実トークン
- 実パスワード
- その他、管理者権限・強い権限を持つ値

### 3.2 クライアントで使ってよいもの

ブラウザ側では以下のみ利用します。

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

Supabase anon key はブラウザ公開前提の値ですが、権限制御はRLSで守る必要があります。  
`service_role` key は絶対にクライアントへ出してはいけません。

## 7. 次の手動チェックポイント

次は、Supabase SQL Editorで以下のSQLファイルを手動実行する段階です。

```txt
supabase/sql/005_external_catch_memos_owner_policies.sql
```

実行前に、SQLの先頭が以下のようになっていることを確認してください。

```sql
-- Owner scoped RLS policy proposal for external_catch_memos.
alter table public.external_catch_memos enable row level security;
```

このSQLでやること:

- `external_catch_memos` のRLSを再度有効化。
- anon/authenticatedの既存権限をrevoke。
- authenticatedに `select, insert, update` のみgrant。
- `owner_id = auth.uid()` の自分のメモだけselect可能にする。
- `owner_id = auth.uid()` の自分のメモだけinsert/update可能にする。
- delete policyは作らない。
- 論理削除は後続のrepository/UI接続で `is_deleted = true` に更新する方針。

実行してはいけないこと:

- service role keyを貼る。
- DB passwordを貼る。
- DB URLを貼る。
- anonにwriteを開放するSQLへ改変する。
- `grant all` を追加する。

## 14. 最終メモ

このプロジェクトは、かなり丁寧に安全側へ寄せて進めています。  
特に外部メモDB保存は「ユーザー入力のwrite」なので、マスターデータreadより危険度が高いです。

次チャットでも、焦ってDB writeを一気に本番化せず、以下の順番を守ってください。

```txt
SQL案作成 → PRレビュー → マージ → 手動SQL実行 → repository接続 → UI確認 → 移行設計
```

```txt
SQL案作成 → PRレビュー → マージ
```

次はこれです。
