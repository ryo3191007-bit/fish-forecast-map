## 8. プロジェクト概要

FishForecastMapは、釣行判断を支援する個人向けの釣り予報マップアプリです。

主な技術:

- Next.js
- React
- TypeScript
- MapLibre GL JS
- Open-Meteo Weather API
- Open-Meteo Marine API
- 国土地理院タイル
- Supabase Auth / Postgres / RLS
- Vercel

主な機能:

- 釣り場マップ。
- 魚種、エリア、日付、キーワード等による絞り込み。
- 釣果実績ベースのSCORE。
- ユーザー自身の釣果記録。
- Supabaseマスターデータ読み取りと静的fallback。
- メール認証。
- ログインユーザー単位の釣果メモDB保存。
- localStorage旧データの明示移行。
- 7日間の時間別天気・風・波・海面水温・潮位参考値・海流。
- 潮回り参考、満潮・干潮参考時刻、気象庁潮位表リンク。
- 通常地図、航空写真、水深・3D地形の切替。

## 10. 最重要の作業ルール

このプロジェクトはGitHub Issue / PRを中心に、ChatGPTとCodexを分担して進めます。

基本フロー:

1. ChatGPTが次の作業を一つだけ整理する。
2. GitHub Issueを作成する。
3. ユーザーがIssue内容を確認する。
4. ユーザーが明示的に`Codex起動OK`と送る。
5. ChatGPTがCodex起動コメントを投稿する。
6. Codexの👀リアクション、Bot返信、Taskリンク等で実起動を確認する。
7. Codexが実装する。
8. PRをChatGPTがレビューする。
9. 問題があれば修正内容を全文で渡す。
10. 問題がなければユーザーがPRをマージする。
11. ChatGPTがマージを確認する。
12. 対応Issueを`completed`で閉じる。

絶対に守ること:

- 必要な作業は一つずつ提示する。
- ユーザーの完了報告前に次タスクを重ねない。
- Codex起動前に`Codex起動OK`を待つ。
- PRの最終マージはユーザーが行う。
- コメント投稿成功だけでCodex起動済みと判断しない。
- Codexの👀等を確認してから起動済みと報告する。
- 修正依頼はコメントID参照だけにせず全文で渡す。
- CI greenだけでマージ可と判断しない。
- 開始ブランチと親コミットを確認する。
- 既存PRを修正する場合は同じブランチ・同じPRを更新する。

## 14. 重要な落とし穴

- `sourceSha256`の文字列だけ公式値に合わせても、値配列が公式由来とは限らない。
- 値配列SHAと変換手順の両方が必要。
- pixel-centreをbounds端の格子点として扱うと半セルずれる。
- 見栄え改善のために水深へ人工ノイズを追加しない。
- TID 17はdirect系。
- TID 127はnodataで、陸セル0とは別。
- nodataを割合の分母へ入れない。
- GEBCO失敗時にsource追加だけではlayerは切り替わらない。
- fallback layerとterrainを明示的に切り替える。
- ETOPOも失敗した場合の最終fallbackが必要。
- GSI標準地図の低zoom追加出典を省略しない。
- PR本文がコードより古いままでもCIは成功する。
- Vercel成功だけでPC / スマホ性能確認済みとは言えない。
- Codexコメント投稿成功だけで起動したと判断しない。
- Codexタスク環境はGitHub remoteや`gh`認証を持たない場合がある。
- 別タスクで既存PRを引き継ぐ場合、親コミットを確認する。
- `supabase/sql`を新規migration代わりに使わない。
- localStorageデータを勝手に全件移行・削除しない。
- anon write、`grant all`、物理deleteを追加しない。
- `service_role` keyをクライアントへ出さない。
