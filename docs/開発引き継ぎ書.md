## 1. 現在の結論

FishForecastMapは、糸島西岸〜唐津湾〜伊万里湾〜平戸周辺を対象とする、個人開発・個人利用の釣り向け参考情報Webアプリです。

主な実装済み機能:

- Next.js / React / TypeScript / MapLibre GL JS。
- 通常地図、航空写真、水深・3D地形の3モード。
- 通常地図は国土地理院の淡色地図。
- 地点マーカー、ポップアップ、釣果カード、フィルタ。
- Open-Meteo Weather / Marineによる7日間の環境予報。
- Supabase master readと静的fallback。
- Supabase Auth、本人釣果の登録・編集・論理削除。
- GEBCO_2026 15秒を第一sourceとする参考水深・3D地形。
- 釣り場属性調査Schema v1.1.0と共通プロンプト。
- 唐津東港を対象としたChatGPT / Gemini / Claudeの3AI比較。

本書更新開始前のGitHub状態では、openの機能Issue・PRは0件でした。現在の文書同期作業はIssue #157で扱います。

## 2. 最重要の作業ルール

### 2.1 基本フロー

1. 次の作業を一つだけ整理する。
2. GitHub Issueを作成する。
3. ユーザーがIssue内容を確認する。
4. 実装担当を決める。Codexを使う場合は、ユーザーの明示的な許可を待つ。
5. 1 Issue = 1 Draft PRとする。
6. 差分、CI、Vercel、制約違反をレビューする。
7. 実ブラウザ確認が必要な場合は、ユーザーがPreviewで確認する。
8. Draft解除・マージはユーザーが判断する。
9. マージ確認後、リンクされたIssueを`completed`として閉じる。

### 2.2 禁止事項

- ユーザーの許可なしにPRをマージしない。
- 複数の機能作業を同じIssue・PRへ混在させない。
- 外部情報の利用条件や公式回答を推測で確定しない。
- Codex管理中のPRブランチへ、Codex外からファイルcommitしない。
- CI greenだけでマージ可と判断せず、差分とテスト内容を確認する。
- 実施していないブラウザ確認、性能計測、外部送信を完了扱いしない。

### 2.3 Codexで既存PRを追加修正する手順

既存PRの追加修正は、PR Conversationのトップレベルコメントへ`@codex`と修正要件全文を投稿します。

必須確認:

1. PRのstate、head branch、head SHAを確認する。
2. Codex管理ブランチへ外部commitが入っていないか確認する。
3. 修正要件をPRへ記録する。
4. ユーザーのCodex起動許可を待つ。
5. トップレベルコメントからCodexを起動する。
6. Botの反応またはTask URLを確認する。
7. HEAD SHA更新、追加commit、完了コメントを確認する。
8. 失敗時に勝手に置き換えPRを作らない。

PR Review submissionに`@codex`を書くだけでは起動しない場合があります。リンク元Issueから起動すると、新規タスク・ブランチ・PRになるため、既存PR修正には使用しません。

### 3.2 航空写真

国土地理院の地理院タイルを使用します。通常地図とは独立した表示モードです。

### 3.3 水深・3D地形

現在のsource:

- 第一source: `GEBCO_2026 Grid 15 arc-second`
- データ由来: `GEBCO_2026 TID Grid`
- fallback: `NOAA NCEI ETOPO 2022 60 Arc-Second Bedrock`

現在の主な機能:

- 2D水深色分け。
- 等深線とズーム別密度調整。
- 陰影profile。
- 陰影・等深線toggle。
- MapLibre terrainによる3D表示。
- 高さ誇張1.0〜4.0倍。
- 真上・斜め・低角度の視点preset。
- タップ地点の緯度経度、表示source、参考水深。
- 端末性能、reduced motion、WebGL対応状況に応じた初期2D / 3D制御。
- GEBCO失敗時のETOPO fallback、両方失敗時の通常地図fallback。

Post-MVP-050で、GEBCO由来の濃緑海岸線ライン、緑の陸地マスク、海岸線表示ボタンは削除済みです。水深モードの陸地pixelは透明として扱います。

水深・3D地形は参考表示です。港内、岩礁、根、瀬、正確な海岸線を保証せず、航海・安全判断には使用できません。

詳細正本:

- `docs/BATHYMETRY_AND_3D_TERRAIN_DESIGN.md`
- `docs/COASTAL_BATHYMETRY_DATA_RESEARCH.md`
- `docs/HIGH_RESOLUTION_3D_BATHYMETRY_PLAN.md`

## 6. 環境予報とSCORE

Weather API:

- 気温、天気コード、降水量、降水確率、風速、風向、突風。

Marine API:

- 海面水温、潮位参考値、波高、波向、波周期、海流速度、海流方向。

条件:

- `timezone=Asia/Tokyo`
- `forecast_days=7`
- Weather / Marineを独立取得。
- 片方成功時はpartial表示。
- 30分以内のcacheはfresh。
- 24時間以内のcacheはAPI失敗時fallback。
- 24時間超は使用しない。

環境値は現在のSCOREへ本格反映していません。次のSCOREは、地点固定特性、魚種生態、季節、環境、本人釣果を分離し、加点・減点理由を説明できるルールベース方式から設計します。

## 7. Supabase / DBの状態

master data:

- `fish_species`
- `fishing_spots`
- `source_registry`

未設定、取得エラー、空データ時は静的データへfallbackします。

本人釣果:

- メールOTP / magic linkログイン。
- 未ログイン時はlocalStorage。
- ログイン済みかつDB利用可能時はSupabaseへ保存。
- DB失敗時はlocalStorage fallback。
- owner-scoped。
- 削除は論理削除。
- 旧localStorage記録は自動移行せず、選択した記録だけ明示移行。

新しいDB変更は`supabase/migrations/`を正本とします。旧SQLファイルをmigration代わりに使用しません。
