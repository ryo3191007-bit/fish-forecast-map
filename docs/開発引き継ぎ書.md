## 2. プロジェクト概要

FishForecastMap は、釣行判断を支援する個人向けの釣り予報マップアプリです。

主な技術:

- Next.js
- React
- TypeScript
- MapLibre GL JS
- 国土地理院航空写真タイル
- Open-Meteo Weather API
- Open-Meteo Marine API
- Supabase Auth / Postgres / RLS
- Vercel

主な機能:

- 釣り場マップ。
- 魚種、エリア、日付、キーワード等による釣果表示・絞り込み。
- 釣果実績ベースのSCORE表示。
- ユーザー手入力の外部釣果メモ。
- Supabaseマスターデータ読み取りと静的fallback。
- メールリンク認証。
- ログインユーザー単位の外部メモDB保存。
- localStorageに残る旧メモの明示移行。
- 7日間の時間別天気・風・波・海面水温・潮位参考値・海流表示。
- 選択日の潮回り参考と、Open-Meteo値から抽出した満潮・干潮参考時刻。
- 気象庁公式潮位表へのリンク。

## 3. 最重要の作業ルール

このプロジェクトは、GitHub Issue / PR を中心に、ChatGPT と Codex を分担して進めます。

### 3.1 役割分担

- ユーザー:
  - 最終仕様判断。
  - Codex画面で必要な操作。
  - PRの最終マージ。
  - 本当に必要な場合のみ、手動の外部サービス設定。
- ChatGPT:
  - 方針整理。
  - Issue作成。
  - Codex起動指示。
  - PR差分・CI・制約違反のレビュー。
  - GitHub上のコメント、Issue、PR状態の整理。
  - マージ後のIssue完了クローズ。
- Codex:
  - Issue本文またはCodex入力欄に直接与えた指示に従って実装。
  - テスト、lint、typecheck、build。
  - PR作成または既存ブランチ更新。

### 3.2 基本フロー

1. ChatGPTが次の作業を一つだけ整理する。
2. GitHub Issueを作成する。
3. ユーザーがIssue内容を確認する。
4. ユーザーが明示的に `Codex起動OK` と送る。
5. ChatGPTがCodexを起動するためのGitHubコメントを投稿する。
6. Codexが実装する。
7. ユーザーが必要なCodex画面操作を行う。
8. PRが作成されたら、ユーザーがPR URLをChatGPTへ渡す。
9. ChatGPTが差分、CI、Vercel、制約違反をレビューする。
10. 問題があれば、修正内容を曖昧にせず全文でCodexへ渡す。
11. 問題がなければ、ユーザーがPRをマージする。
12. ChatGPTがマージを確認する。
13. ChatGPTが対応Issueを `completed` でクローズする。

### 3.3 絶対に守ること

- 必要な作業は一つずつ提示する。
- ユーザーの完了報告前に、次の作業を重ねて出さない。
- Codex起動前に、必ずユーザーの `Codex起動OK` を待つ。
- PRの最終マージはユーザーが行う。
- マージ確認後に、対応Issueを `completed` で閉じる。
- 実在しないCodexボタンを案内しない。
- Codex画面をGitHub画面と取り違えない。
- CodexがGitHubコメントを直接読めると決めつけない。
- 修正依頼で「最新コメントを見て」「コメントIDを見て」だけを送らない。
- Codexに修正させる場合は、修正対象、実装方法、完了条件、テスト条件をCodex入力欄へ全文で渡す。
- PRを別タスクへ置き換える場合は、開始ブランチ・親コミットを必ず検証する。
- CI成功だけでマージ可と判断せず、実際に必要なテストがCIで走っているか確認する。

## 4. セキュリティと外部情報のルール

### 4.1 秘密情報

GitHub、Issue、PR、チャット、ログへ貼らないもの:

- Supabase `service_role` key
- Supabase DB password
- Supabase DB URL / Postgres connection string
- `.env.local`
- 実トークン
- 実パスワード
- 管理者権限を持つ値

ブラウザ側で利用してよいもの:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

anon keyは公開前提ですが、権限制御はRLSで守ります。  
`service_role` keyはクライアントへ出しません。

### 4.2 外部釣果情報

- スクレイピングしない。
- 自動収集しない。
- 定期実行しない。
- AIによる外部記事解析を追加しない。
- 外部メモはユーザーが手動入力する。
- 保存対象は要約メタデータのみ。
- 記事本文、画像、全文コメント、プロフィール情報を保存しない。
- 現在 `crawlPolicy === "allowed"` のサイトはない。

保存可能な要約項目の例:

- 魚種
- 釣果日
- エリア
- 推定地点
- 釣法
- 数
- サイズ
- 参照元名
- 参照元URL
- acquisitionMethod
- confidence

### 4.3 潮汐・安全情報

- Open-Meteoの `sea_level_height_msl` は「潮位参考値」。
- 公式の満潮・干潮時刻ではない。
- 航海、立入可否、避難、その他の安全判断には使わない。
- 潮回りは月齢からの簡易算出であり、商用潮見表と境界日が異なる可能性がある。
- 気象庁の潮位表はリンクで参照するだけで、スクレイピング・複製・保存しない。
- 波や風が低いことを「安全」と断定しない。

### 6.3 外部釣果メモ

現在の基本動作:

- 未ログイン時はlocalStorage。
- ログイン済みかつDB read成功時は、新規メモをSupabaseへ保存。
- DB失敗時はlocalStorage fallback。
- DB由来メモはowner-scopedで読み書き。
- 削除は物理削除ではなく `is_deleted = true`。
- 論理削除は `soft_delete_external_catch_memo` RPCを使用。
- RPCはauthenticated userの自分の行だけを対象にする。
- 旧localStorageメモは自動移行しない。
- ユーザーが選択したメモだけ明示的にSupabaseへ移行する。
- 移行成功確認後も、失敗分を失わない。
- tombstone-only状態は、ユーザー向けの「未移行データあり」として表示しない。

主要ファイル:

- `src/lib/externalCatchMemoStorage.ts`
- `src/lib/externalCatchMemoMapper.ts`
- `src/lib/externalCatchMemoRepository.ts`
- `src/hooks/useExternalCatchMemos.ts`
- `src/components/ExternalCatchMemoSection.tsx`
- `src/domain/externalCatch.ts`

### 6.4 DB migration運用

今後のDB変更は、次を正本にします。

```txt
supabase/migrations/<timestamp>_<description>.sql
```

`supabase/sql/001`〜`007`は、過去の手動構築・設計・参照用ファイルです。  
新しいDB変更を `supabase/sql` へ追加して、手動SQL Editor実行を標準運用に戻さないでください。

### 7.3 キャッシュと失敗時動作

- localStorageのバージョン付きキャッシュ。
- 30分以内はfresh cache。
- 24時間以内はAPI失敗時fallback候補。
- 24時間超は利用しない。
- JSON破損、旧version、構造不正、localStorage例外で画面を落とさない。
- Weather / Marineの全数値項目が欠損したオブジェクトを有効扱いしない。
- availabilityフラグと実データが矛盾するキャッシュを拒否。
- 地点切替時はAbortと `isActive` ガードを併用し、古い応答で新地点を上書きしない。
- タイムアウトやHTTP失敗では有効キャッシュへfallback。
- 呼出元Abortはfallbackへ変換せず上位へ伝播。

### 7.4 時刻と潮位判定の注意

- Open-Meteoが返すoffsetなし時刻はAsia/Tokyoとして解釈する。
- `Date.parse`へ直接渡さず、`parseForecastTime`を使う。
- Weather / Marineの `time` 配列は、元indexを保持して値配列と対応させる。
- `time`途中の `null` でindexを詰めない。
- 満潮・干潮参考は、前日末尾と翌日先頭の有効潮位も比較に使う。
- 返すイベントは選択日に属する時刻だけ。

### 7.5 SCOREとの関係

現在、環境値はSCOREへ反映していません。

禁止:

- `forecastScore.ts`へ環境値を直接追加。
- API失敗時にSCOREを下げる。
- キャッシュ状態でSCOREを変える。
- 潮位参考値を釣果スコアや安全スコアへ変換する。

環境補正を導入する場合は、必ず別Issueで設計・実装してください。

## Post-MVP-035 自分の釣果記録UI

- 手入力釣果は外部釣果サイトのメモではなく、ユーザー自身が実際に釣った釣果記録として扱う。
- 「釣果を登録」モーダルは入力フォームのみを表示し、登録済みカード一覧・localStorage移行一覧・大きな保存先説明ブロックを含めない。
- 編集は釣果一覧カードごとの明示的な「編集」ボタンから同じフォームモーダルを開く。削除は編集時のみ表示し、確認後に実行する。
- 情報元URL・情報元・信頼度は画面入力から外す。新規登録時だけ `user-self-report` / `本人の釣果` / `manual` / `high` を内部固定値として使い、既存レコード編集時は既存source値を維持する。
- `ExternalCatchMemo` 型、repository、`external_catch_memos` テーブル名、既存localStorage keyが残るのは後方互換のため。
- localStorage明示移行は自動移行・自動削除・一括upsertにせず、登録モーダル外のコンパクトな別導線で維持する。
