## 1. 現在の結論

FishForecastMapは、糸島西岸〜唐津湾〜伊万里湾〜平戸周辺を対象とする、個人開発・個人利用の釣り向け参考情報Webアプリです。

- 承認済み魚種別名を、安定した内部魚種IDへ正規化する基盤。
- GEBCO_2026 15秒を第一sourceとする参考水深・3D地形。

主要実装は`main`へ反映済みです。ただしIssue #195はopenのままです。

実装内容:

- 気象庁防災情報XMLのPULL型Atomフィードをサーバー側で取得。

- 選択地点・選択日時に対する`blocked / clear / unknown / out-of-range`判定。
- JMAとOpen-Meteoを統合したfail-closed安全ゲート。
- `blocked`または`unknown`時は総合点を表示しない。
- 取得失敗、古い電文、schema不一致を安全扱いしない。
- API route: `GET /api/jma-warnings`
- fixtureを使ったoffline test。

主要ファイル:

### 2.2 Issue #239 / PR #242: 安全ゲート表示と地点詳細の暫定表現整理

安全判定ロジックは変えず、通常UIの表示条件を整理しました。

表示仕様:

- `blocked`: 対象区域、現象、電文、発表時刻、対象時間帯、出典を含む詳細を表示。
- `unknown`: 内部reasonを出さず、総合点を表示しない旨の短文を1つだけ表示。
- `clear`: JMA詳細パネルを表示しない。
- `out-of-range`: JMA詳細パネルを表示しない。既存の統合判定へ従う。
- loading中: 取得失敗の文言を表示しない。
- 想定外state: 安全側の`unknown`相当へ倒す。

地点詳細の通常UIでは、調査・内部管理用の次の表現を除去します。

- `候補`
- `候補（現行未確認）`
- `候補（low / medium / high）`
- `（low / medium / high）`
- 全角・半角括弧、空白差分

例:

- `照明候補（現行未確認）` → `照明`
- `頭上電線注意候補（low）` → `頭上電線注意`
- `手すり候補（low）` → `手すり`

source、内部note、confidence等の証跡は削除せず、通常UIへだけ露出させません。赤い`信憑性: 低`バッジは利用者向け情報として残します。

## 4. 地図・水深・環境予報

### 4.1 通常地図

- 国土地理院淡色地図。
- tile: `https://cyberjapandata.gsi.go.jp/xyz/pale/{z}/{x}/{y}.png`
- `tileSize: 256`
- `maxzoom: 18`
- 航空写真、水深・3D地形、marker、popup、自動fitは独立して維持。

### 4.2 水深・3D地形

現在のsource:

- 第一source: `GEBCO_2026 Grid 15 arc-second`
- fallback: `NOAA NCEI ETOPO 2022 60 Arc-Second Bedrock`

主な機能:

- 2D水深色分け。
- 等深線。
- 陰影。
- MapLibre terrainによる3D表示。
- 高さ誇張。
- 視点preset。
- タップ地点の緯度経度、表示source、参考水深。
- GEBCO失敗時のETOPO fallback。
- 両方失敗時の通常地図fallback。

水深・3D地形は参考表示です。港内、岩礁、根、瀬、正確な海岸線を保証せず、航海・安全判断には使用できません。

### 4.3 Weather / Marine

Weather API:

- 気温。
- 天気コード。
- 降水量・降水確率。
- 風速・風向・突風。
- 日の出・日の入り。

Marine API:

- 海面水温。
- 潮位参考値。
- 波高・波向・波周期。
- 海流速度・海流方向。

条件:

- `timezone=Asia/Tokyo`
- `forecast_days=7`
- Weather / Marineを独立取得。
- 片方成功時はpartial表示。
- offsetなし時刻をAsia/Tokyoとして解釈する。
- time配列をfilterしてindexを詰めない。
- 全数値nullを取得成功と扱わない。
- API失敗、partial、古いcacheをSCORE低下へ直接使わない。
- 潮位参考値、波、風から「安全」と断定しない。

## 7. 外部釣り情報と地点調査のpolicy

- scraping、自動巡回、定期取得を行わない。
- 民間siteの本文、画像、地図、魚種一覧、comment、profileを保存・転載しない。
- 民間siteは地点候補や追加確認事項を知る入口に限定する。
- 公式・公的情報または複数の独立sourceで再確認する。
- 同じ原典・転載・同一運営元を独立sourceとして重複countしない。
- sourceが直接支えない属性を`supports`へ入れない。
- 一般立入、釣り可否、足場、駐車、トイレ、常夜灯を地点種別だけから推定しない。
- 魚種はobservedとexpectedを分離する。
- 不明値は`unknown / low / supportingSourceIds: []`を維持する。
- 外部siteの地点DB、説明文、写真、評価、座標一覧をそのまま複製しない。
- 地点名、位置、属性は公的資料、地理院地図、自治体、港湾・漁港資料等から独自に整理する。
- 危険地点、立入禁止、小場所保護へ配慮する。
- 地点の代表座標を、実釣位置や安全な立入地点として保証しない。

## 8. 最重要の作業rule

### 8.1 基本flow

1. GitHub正本を確認する。
2. 次の作業を一つだけ整理する。
3. 1 Issue = 1 PRとする。
4. 実装担当と書き込み主体を明示する。
5. Codexを使う場合は、利用者の明示許可を待つ。
6. Codexへ渡す要件、対象外、完了条件、test条件を最初の1回へ統合する。
7. 最新`main`から新規branchを作る。
8. PR作成後は最新HEAD、変更file、全diff、CI、Vercel、制約違反を確認する。
9. 実browser確認が必要な場合はVercel Previewで確認する。
10. PRのmergeは利用者が判断する。
11. merge確認後に限り、利用者の明示指示に従ってIssueを`completed`として閉じる。
12. 大きな状態変化があれば、本ファイルを更新する。

### 8.2 禁止事項

- 利用者の明示判断なしにCodexを起動しない。
- 利用者の明示判断なしにPRをmergeしない。
- 利用者の明示判断なしにIssueをcloseしない。
- 利用者の明示判断なしにremote DBへ適用しない。
- 複数の独立機能を同じIssue・PRへ混在させない。
- 同じPR、同じbranch、同じCodex作業へ、分割したCodex指示を重ねない。
- 同じPRへ2回目のtop-level `@codex`指示を投稿しない。
- CI greenだけでmerge可能と判断しない。
- Codexの「完了」「PR更新」報告だけで実装成功と判断しない。
- 実施していないbrowser確認、性能計測、外部送信、DB反映を完了扱いしない。
- Codex Cloud内のcommitや`make_pr` metadataをGitHub反映済みと扱わない。
- 利用者本人が作成したPRをCodex作成と記録しない。
- GitHub上の対象file更新依頼を、local file出力で代替しない。
- `main`へ直接fileを作成、更新、削除しない。
- force push、reset、履歴改変を利用者の明示許可なしに行わない。
- 不明な値をAI推測で`confirmed`にしない。
- placeholder URLや架空sourceを正式な根拠として保存しない。
- 未承認のbinary fileを作成、保存、変更、commitしない。

### 8.3 実行主体の記録

branch、commit、PRを誰が作成したかを分離して記録します。

- Codex Cloud: 隔離環境内の編集、test、commit候補。
- ChatGPTのGitHub接続: GitHub API経由で実際に行ったIssue、branch、commit、PR操作。
- 利用者: 利用者本人が手動で行ったpush、PR作成、merge、Issue close等。
- GitHub上で確認できない主体・経路: `未確認`。

GitHub上のbranch、HEAD SHA、PR番号、stateを取得するまで、GitHub反映完了と報告しません。
