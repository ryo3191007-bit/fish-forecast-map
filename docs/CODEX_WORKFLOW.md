# Codex作業フロー

## 目的

このプロジェクトでは、GitHub IssueとPull RequestをCodexへの作業指示インターフェースとして利用します。

ChatGPTが司令塔/プロジェクトリードとして作業を整理し、Codexが実装担当として作業します。

## 標準フロー

1. ChatGPTが仕様ドラフト、懸念点、確認事項をこのチャットに提示する。
2. ユーザーが仕様更新を承認する。
3. ChatGPTがGitHub上の仕様書を更新する。
4. ChatGPTがGitHub Issueを作成、または既存Issueを更新する。
5. ユーザーがCodex起動を承認する。
6. ChatGPTまたはユーザーがCodex起動コメントを投稿する。
7. CodexがPull Requestを作成する。
8. ChatGPTがPR内容とCI結果をレビューする。
9. 修正が必要な場合、ChatGPTが追加指示案を提示する。
10. ユーザー承認後に修正指示を出す。
11. ユーザーが最終承認してからマージする。

## タスクサイズのルール

1タスク = 1 Issue = 1 Pull Request とします。

無関係な機能追加、リファクタリング、UI変更、インフラ変更を1つのPRにまとめないでください。

## 仕様書更新ルール

- 仕様書、AGENTS.md、README、設計ドキュメントの作成・更新は、必ずこのチャットでユーザー確認後にGitHubへ反映する。
- 仕様更新時は、懸念点と未確定事項を必ず提示する。
- ユーザーが「反映OK」と明示した場合のみGitHubへ反映する。

## Codex起動ルール

- Issue作成はChatGPTが行ってよい。
- Codex起動はユーザー承認後に行う。
- ユーザー承認前にCodex起動コメントを投稿しない。
- 誤って起動コメントを投稿した場合は、コメントを無効化し、ユーザーへ報告する。

## Issueの形式

実装Issueには以下を含めます。

- 目的。
- 背景。
- 対象範囲。
- 対象外。
- 実装メモ。
- 受け入れ条件。
- Codex起動前の注意。

## PRの期待内容

各PRには以下を含めます。

- 変更内容。
- 変更理由。
- UI変更がある場合は可能な範囲でスクリーンショット。
- 動作確認方法。
- 既知の制約。
- ドキュメント更新の有無。

## レビュー方針

ChatGPTは以下をレビューします。

- スコープがIssueから逸脱していないか。
- ドキュメントと実装が一致しているか。
- UI/UXに大きな違和感がないか。
- データ利用ポリシーに反していないか。
- コスト方針に反していないか。
- CI/buildが通っているか。

Claude/Geminiなどの外部AIレビューは、重要な設計、データポリシー、UXレビューで価値がある場合に限定して利用します。

## トークン節約方針

- Issueは小さく、明確にする。
- ドキュメントを正本として扱う。
- 毎回プロジェクト全体の説明を繰り返さない。
- 初期段階ではモックデータと小さなPRを優先する。
- Codexには実装を任せ、広すぎる企画相談はさせない。
- 外部AIレビューは、効果が見込める場合に限定する。

## Codexへの指示例

```text
このIssueを小さなPRで実装してください。作業前にAGENTS.mdと関連ドキュメントを読んでください。変更範囲はこのIssueに限定してください。可能な限りlint/typecheck/buildが通る状態にしてください。
```
