# ロードマップ

最終更新: 2026-07-18 JST

## Phase 0: プロジェクト初期化

- GitHubリポジトリを作成する。
- GitHub / Codex連携の作業フローを整える。
- README、AGENTS.md、docs配下の初期ドキュメントを作成する。
- 最初の実装Issueを作成する。

## Phase 1: MVPアプリの土台作成

- Next.js + TypeScriptプロジェクトを作成する。
- 基本レイアウトとナビゲーションを追加する。
- MapLibre GL JSを導入する。
- ドメイン型とサンプルデータを追加する。
- 基本CIを追加する。

## Phase 2: 釣果情報UI

- 釣果情報一覧・カードを作成する。
- 魚種フィルタを追加する。
- 0〜100点の釣れそう度SCOREと根拠を表示する。

## Phase 3: 地図UI

- 糸島西岸から唐津湾、伊万里湾、平戸方面までの代表地点を表示する。
- 地点マーカー、ポップアップ、カードを表示する。
- フィルタと地図表示を連動させる。
- 通常地図には国土地理院の淡色地図、航空写真には地理院タイルを使用する。

## Phase 4: 潮汐・天気・水温連携

- Open-Meteo Weather / Marineによる7日間の環境予報を表示する。
- 潮位参考、波、風、水温、海流を安全判断には使用しない。
- 環境値をSCOREへ反映する場合は、別Issueで説明可能なルールを設計する。

## Phase 5: 水深・3D海底地形表示

- GEBCO_2026 15秒を第一source、ETOPO 2022 60秒をfallbackとして使用する。
- 2D水深色分け、等深線、陰影、MapLibre terrain、参考水深を提供する。
- 高精細データは、利用条件とWeb配信可否を公式に確認できた場合だけPoCへ進む。
- 参考表示であり、航海・安全判断には使用しない。

## Phase 6: 釣果予測SCOREの高度化

- 地点の固定特性、魚種生態、季節、環境、本人釣果を分離して扱う。
- 最初は説明可能なルールベースのSCORE v2を設計する。
- 魚種別・場所別の重みと、スコア理由を表示する。

## Phase 7: 釣果・地点情報の取り込み

- 民間釣りサイトは地点候補を知る入口に限定する。
- 記事本文、画像、地図ピン、魚種一覧の転載やデータベース再現を行わない。
- 公式・公的情報、許可を得た情報源、本人入力を優先する。
- 自動巡回、スクレイピング、定期取得は行わない。

## Phase 8: Supabase設計・連携

- master data、認証、本人釣果の読み書きは実装済み。
- 新しいDB変更はmigrationを正本とする。
- 調査値の本番反映や編集UIは、承認・履歴方針を決めてから追加する。

## 次の優先順（3D・水深以外）

### 1. 海しる公式回答の記録と利用可否判断

回答が届いたら、利用可能な項目、加工・公開・保存条件、出典・注意書き、追加確認事項を整理します。回答前に仕様や許諾を推測しません。

### 2. 調査済み地点の本番適用確認と追加調査方針

Issue #178の17地点はリポジトリ上の安全な最小値反映準備段階であり、remote Supabaseへの適用確認、プレビュー目視、未確定代表座標の追加確認は後続Issueで扱います。新規調査では `district / facility / access_point` の単位を明示し、魚種・釣法は直接根拠がある場合だけ採用します。

### 4. 魚種生態マスター

魚種ごとの季節、水温、水深、底質、潮流、外海・河口との相性、昼夜、釣法を、根拠と更新日を持つ形で整理します。

### 5. 説明可能なSCORE v2

地点固定特性、魚種生態、選択日時の環境、本人釣果を分離し、各加点・減点理由を表示できるルールベースSCOREを設計します。

### 6. 調査値の確認・編集UI

AI調査値を無条件に正解扱いせず、採用、修正、不明化、根拠確認、期限切れ確認ができる管理用UIを検討します。

各作業は別Issueとし、1 Issue = 1 Draft PR、最終マージはユーザーが判断します。
