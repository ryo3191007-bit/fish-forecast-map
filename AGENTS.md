# AGENTS.md

## プロジェクト概要

このリポジトリは、個人利用向けの釣果予測マップアプリを開発するためのリポジトリです。

目的は、福岡県糸島市西岸から唐津湾、伊万里湾、平戸方面までの陸っぱり釣り情報を地図上で可視化し、魚種、場所、釣法、釣れそう度スコアを確認できるようにすることです。

対象は、堤防・磯場などの陸っぱり釣りです。釣法はルアー・エサの両方を扱います。

MVPでは機能を小さく保ち、低コストで安全に検証できる範囲に限定していました。Post-MVPでは手入力釣果登録、Supabase Auth、ログインユーザー単位のDB保存/fallbackが段階的に実装済みです。

## 役割分担

- ユーザー: 最終承認、方向性判断、プロダクト方針の決定を行う。
- ChatGPT: 司令塔/プロジェクトリードとして、要件整理、タスク分解、Issue作成、レビュー、マージ判断の補助を行う。
- Codex: 実装担当として、GitHub IssueとPull Requestを単位に作業する。
- Claude/Geminiなどの外部AI: 必要な場合のみ、重要な設計、データポリシー、UXレビューの補助として利用する。

## 優先順位

作業では以下の順に優先します。

1. 既存のChatGPT Plus利用範囲を中心に、月額コストを抑えること。
2. トークン消費を抑えること。
3. 品質を維持すること。
4. 生産性を高めること。

明示的な承認がない限り、有料API、有料ホスティング、有料地図サービス、有料モデル推論は導入しません。

## 作業ルール

- 1タスク = 1 GitHub Issue = 1 Pull Request とします。
- 無関係な変更を1つのPRにまとめないでください。
- 作業前に、このファイルと `docs/` 配下の関連ドキュメントを確認してください。
- 実装とドキュメントの内容を一致させてください。
- 実装とドキュメントが矛盾する場合は、PR内で確認するか、必要に応じてドキュメントも更新してください。
- PRはレビューしやすい小さな単位にしてください。
- 可能な限り、lint、typecheck、build、testが通る状態にしてください。

## 仕様書・Issue・Codex起動の運用

- 仕様書、AGENTS.md、README、設計ドキュメントの作成・更新は、必ずこのチャットでユーザー確認後にGitHubへ反映します。
- 仕様書作成・更新時は、懸念点と未確定事項を必ずユーザーに伝えます。
- Issue作成はChatGPTが行ってよいです。
- Codex起動はユーザー承認後に行います。
- PRマージは必ずユーザーの最終承認後に行います。
- ユーザー承認前に、Codexへ実装開始を促すコメントを投稿しません。

## MVP v0.1 の記録とPost-MVPの現状

以下はMVP v0.1当時の制約を含みます。現在のPost-MVPでは、一部は実装済みの現状に合わせて読み替えます。

- 最初はモックデータを使用します。
- 対象エリアは、福岡県糸島市西岸から唐津湾、伊万里湾、平戸方面までとします。
- 地図はMapLibre GL JSを第一候補にします。
- 地図上にモック釣果地点のマーカーを表示します。
- MVP時点では釣果登録フォーム、ユーザー自身の釣果登録機能は実装しませんでした。Post-MVPでは、手入力釣果登録モーダル、Supabase Auth、ログインユーザー単位のSupabase DB保存、未ログイン・未設定・DBエラー時のlocalStorage fallbackが実装済みです。
- 外部の釣果サイトの自動取り込みは実装しません。現在も自動取り込み、スクレイピング、定期アクセス、AI解析は行いません。
- 3D海底地形表示は実装しません。
- 有料APIは実装しません。
- 複雑な機械学習による予測は実装しません。
- 釣れそう度は、0〜100点の説明可能な簡易ルールベースのスコアにします。
- 釣れそう度の根拠を表示します。
- 釣果情報には出典情報を持たせます。
- 現在の釣果一覧は `acquisitionMethod: "manual"` の手入力釣果のみを表示します。
- モック釣果は地図・地点評価・既存SCORE用の参考データとして維持します。
- 地図、地点評価、`SCORE` 補正など一覧以外の既存挙動では、必要に応じて既存の外部メモデータを参照し、一覧用のmanual限定と混同しません。
- 認証操作はヘッダーのログインボタンから開くモーダルに集約します。
- 画面下部の外部参考リンクは参考閲覧用であり、アプリによる自動取得ではありません。

## 推奨技術スタック

- フロントエンド: Next.js + TypeScript。
- スタイリング: Tailwind CSS またはシンプルな CSS Modules。
- 地図: MVPでは MapLibre GL JS を第一候補とします。
- データベース: Supabase/PostgreSQL。MVPではモックデータのみでしたが、Post-MVPではマスターデータread、Auth、手入力釣果のowner-scoped DB保存/fallbackまで実装済みです。
- CI: GitHub Actionsで install、lint、typecheck、buildを確認。
- 将来の3D表示: Three.js、deck.gl、CesiumJSのいずれかを検討。

## コード方針

- ドメインエンティティにはTypeScriptの型を定義してください。
- UIコンポーネントは小さく、読みやすく保ってください。
- UIとドメインロジックはできるだけ分離してください。
- コメントは、意図が分かりづらい処理を補足する場合に限定してください。
- シークレット情報をハードコードしないでください。
- `.env` ファイルはコミットしないでください。

## ドメイン用語

- 釣果情報: 実際に釣れた魚、場所、日時、釣り方などの記録。
- 釣り場: 釣果があった地点、または予測対象となる地点・海域。
- 魚種: アジ、サバ、イワシ、青物、シイラ、ヒラメ、マゴチ、シーバス、イカ類、チヌ、真鯛、キス、根魚など。
- 釣れそう度スコア: 釣果を保証するものではなく、参考情報として表示する0〜100点の説明可能なスコア。
- 出典: 釣果情報の元になったURL、サイト名、手入力情報など。

## データ利用・法務上の制約

釣具店、ブログ、SNS、ニュースサイトなどの釣果情報は慎重に扱います。

- 各サイトの利用規約、robots.txt、著作権、アクセス負荷に配慮してください。
- MVPでは外部サイトの取り込みは行わず、モックデータのみを使います。
- 将来の取り込みでは、公式API、RSS、許可を得た情報源、ユーザー提供のURL/本文を優先します。
- 保存するのは抽出した事実情報と出典URLを中心にします。
- 出典を明示してください。
- アプリ上で釣果を保証する表現をしないでください。
- 将来公開する場合は、釣り場座標の丸めや非公開化を検討してください。

## Pull Requestの期待内容

各PRには以下を記載してください。

- 変更内容。
- 変更理由。
- UI変更がある場合は可能な範囲でスクリーンショット。
- 動作確認方法。
- 既知の制約。
- ドキュメント更新の有無。

## 明示的な承認なしに禁止すること

- 有料APIの導入。
- 高頻度の自動取り込み。
- 第三者サイトの釣果記事本文の再掲載。
- ユーザーの正確な現在地を同意なく収集すること。
- シークレット情報をリポジトリに保存すること。
- Issueなしの大規模な設計変更。
- ユーザー承認なしのCodex起動。

## DB migration運用ルール

- 今後のDB変更は `supabase/migrations/<timestamp>_<description>.sql` を正本とし、Remote SQL EditorやTable Editorで直接スキーマ変更しないでください。
- 既存の `supabase/sql/*.sql` は参照資料または手動適用済みSQLの記録として扱い、無条件に本番へ再実行しないでください。
- 1 migrationは目的を絞り、forward-onlyを基本にします。PRにはrollback方法または復旧方針、アプリ変更との適用順序を記載してください。
- 後方互換性が必要なDB変更は expand → migrate → contract の段階適用を検討してください。
- `DROP TABLE`、`DROP COLUMN`、大量更新、不可逆な型変更、RLS無効化、anon write拡大などの破壊的または高リスクな変更は、ユーザー承認前にマージしないでください。
- production deploy workflowは初回remote履歴同期完了まで `supabase db push` を実行しない安全ゲートを維持してください。
